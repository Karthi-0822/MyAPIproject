Mocking API responses with page.route() in Playwright — complete guide covering intercept, fulfill, modify, and real-world patterns

How page.route() intercepts requests

page.route() sits between the browser and the network. It intercepts matching requests before they leave the browser and lets you decide what happens next.

Without mocking — real network call
browser → network request → real API server → real response → browser
With page.route() — intercepted
browser → page.route() → route.fulfill() → browser (gets mock)
browser → page.route() → route.continue() → real API → browser

page.route() is a UI test tool — it intercepts requests made by the browser page, not by Playwright's own request fixture. Use it in tests that open a browser page and you want to control what the page's fetch/XHR calls receive, without hitting a real backend.

Three things you can do with an intercepted request
ActionMethodWhen to use
Return a fake responseroute.fulfill()Simulate any response — success, error, slow — without a real server
Let it through unchangedroute.continue()Pass-through — selectively mock only some requests
Modify then pass throughroute.fetch() + route.fulfill()Real response but tweak one field or header

page.route() only works with a browser page. It cannot intercept requests made via Playwright's request fixture or via Node.js code running outside the browser. If you want to mock in pure API tests without a browser, use a library like msw (Mock Service Worker) instead.

Basic mock — fulfill with fake data

Intercept a URL pattern and return exactly what you want. No real server needed.

Simple JSON mock
import { test, expect } from '@playwright/test';

test('shows user profile from mocked API', async ({ page }) => {

  // Intercept before navigating — route is active immediately
  await page.route('**/api/users/1', async route => {
    await route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify({
        id: 1,
        name: 'Alice',
        email: 'alice@example.com',
        role: 'admin',
      }),
    });
  });

  // Navigate — the page's fetch call hits our mock, not the real server
  await page.goto('/profile');

  // Assert the UI rendered the mocked data
  await expect(page.getByText('Alice')).toBeVisible();
  await expect(page.getByText('alice@example.com')).toBeVisible();
  await expect(page.getByTestId('role-badge')).toHaveText('admin');
});
URL pattern matching — the glob rules
// ** matches any number of path segments
page.route('**/api/users', handler);        // any host, path ends in /api/users
page.route('**/api/users/*', handler);      // any user id
page.route('https://api.example.com/**', handler); // entire domain

// Regex for precise matching
page.route(/\/api\/users\/\d+/, handler);   // /api/users/123 but not /api/users/abc

// Selective — only mock one endpoint, pass everything else through
page.route('**', route => {
  if (route.request().url().includes('/api/users')) {
    route.fulfill({ status: 200, body: JSON.stringify([]) });
  } else {
    route.continue();  // everything else hits the real server
  }
});

Set up routes before page.goto(). The route handler must be registered before the page navigates — otherwise the page's initial fetch calls fire before Playwright can intercept them. Always await page.route() first, then await page.goto().

Error scenarios — simulate failures

The most valuable use of mocking — testing how your UI handles 404s, 500s, network failures, and slow responses. Impossible to test reliably without mocking.

404 — resource not found
test('shows "not found" message when user does not exist', async ({ page }) => {
  await page.route('**/api/users/999', route => route.fulfill({
    status: 404,
    contentType: 'application/json',
    body: JSON.stringify({ error: 'User not found' }),
  }));

  await page.goto('/users/999');
  await expect(page.getByTestId('error-message')).toHaveText('User not found');
  await expect(page.getByTestId('profile-card')).not.toBeVisible();
});
500 — server error
test('shows error banner when server fails', async ({ page }) => {
  await page.route('**/api/posts', route => route.fulfill({
    status: 500,
    contentType: 'application/json',
    body: JSON.stringify({ error: 'Internal server error' }),
  }));

  await page.goto('/posts');
  await expect(page.getByRole('alert')).toBeVisible();
  await expect(page.getByTestId('post-list')).not.toBeVisible();
});
Network failure — connection refused
test('shows offline message on network failure', async ({ page }) => {
  // abort() simulates the request failing at the network level
  await page.route('**/api/**', route => route.abort('failed'));

  await page.goto('/dashboard');
  await expect(page.getByText('Unable to connect')).toBeVisible();
});
Slow response — loading states
test('shows loading spinner during slow request', async ({ page }) => {
  await page.route('**/api/posts', async route => {
    // Delay the response by 2 seconds
    await new Promise(resolve => setTimeout(resolve, 2000));
    await route.fulfill({
      status: 200,
      body: JSON.stringify([]),
    });
  });

  await page.goto('/posts');

  // Spinner must appear immediately
  await expect(page.getByTestId('loading-spinner')).toBeVisible();

  // After response arrives, spinner must disappear
  await expect(page.getByTestId('loading-spinner')).not.toBeVisible();
});

These four error scenarios — 404, 500, network abort, and slow response — are the ones most commonly missing from UI test suites, and the ones most commonly broken in production. Mocking makes all four trivial to test consistently.

Modify a real response

Fetch the real API response, tweak one field, and return the modified version. Tests against real data with controlled edge cases.

route.fetch() + modify + fulfill
test('shows admin banner when role is admin', async ({ page }) => {

  await page.route('**/api/users/me', async route => {
    // Step 1 — make the real request
    const realResponse = await route.fetch();

    // Step 2 — parse the real body
    const body = await realResponse.json();

    // Step 3 — change just what we need
    body.role = 'admin';

    // Step 4 — return the modified response
    await route.fulfill({
      response: realResponse,        // inherit real status + headers
      body: JSON.stringify(body), // override just the body
      contentType: 'application/json',
    });
  });

  await page.goto('/dashboard');
  await expect(page.getByTestId('admin-banner')).toBeVisible();
});

Passing response: realResponse to fulfill() inherits the real response's status code, headers, and cookies — you only override what you explicitly set. This means your mock stays realistic without having to replicate every header manually.

Inject extra items into a list response
test('handles large list correctly', async ({ page }) => {
  await page.route('**/api/posts', async route => {
    const real = await route.fetch();
    const posts = await real.json();

    // Inject 50 extra items to test pagination/virtual scroll
    const extras = Array.from({ length: 50 }, (_, i) => ({
      id: 9000 + i, title: `Injected post ${i}`, userId: 1,
    }));

    await route.fulfill({
      response: real,
      body: JSON.stringify([...posts, ...extras]),
    });
  });

  await page.goto('/posts');
  await expect(page.getByTestId('pagination')).toBeVisible();
});

Real-world patterns

Patterns that appear in production test suites — counting calls, unrouting, and loading mocks from fixture files.

Count how many times an endpoint was called
test('search debounces — fires only one request per typed word',
  async ({ page }) => {
    let callCount = 0;

    await page.route('**/api/search**', async route => {
      callCount++;
      await route.fulfill({
        status: 200,
        body: JSON.stringify({ results: [] }),
      });
    });

    await page.goto('/search');
    await page.getByPlaceholder('Search...').fill('playwright');

    // Wait for debounce to settle
    await page.waitForTimeout(500);

    // Typing 10 chars should fire only 1 debounced request
    expect(callCount).toBe(1);
  }
);
Load mock data from a JSON fixture file
// tests/fixtures/users.json
// [{ "id": 1, "name": "Alice", "role": "admin" }, ...]

test('renders all users from fixture file', async ({ page }) => {
  await page.route('**/api/users', async route => {
    await route.fulfill({
      status: 200,
      path: 'tests/fixtures/users.json', // serve file directly
    });
  });

  await page.goto('/users');
  await expect(page.getByTestId('user-row')).toHaveCount(3);
});
Unroute — remove a mock mid-test
test('mock only the first call, then use real API', async ({ page }) => {
  const handler = async (route: Route) => {
    await route.fulfill({ status: 200, body: JSON.stringify({ cached: true }) });
  };

  await page.route('**/api/data', handler);
  await page.goto('/dashboard');

  // Remove the mock — subsequent calls hit the real server
  await page.unroute('**/api/data', handler);

  // Trigger a refresh — this now hits the real API
  await page.getByRole('button', { name: 'Refresh' }).click();
  await expect(page.getByTestId('data-panel')).toBeVisible();
});

The path: option in route.fulfill() is one of the most underused features. It serves a file directly from disk as the response body — no fs.readFileSync(), no JSON.parse/stringify, no manual content-type guessing. Playwright infers the content-type from the file extension automatically.
