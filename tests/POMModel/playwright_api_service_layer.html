Page Object Model service layer for Playwright API tests — complete guide from base client to domain services, fixtures, and full test file

Why a service layer — and what it looks like

Without a service layer, HTTP details bleed into every test. With one, tests read like business scenarios.

Without service layer — HTTP details everywhere
test('user can update their profile', async ({ request }) => {
  // Login — repeated in every test file
  const loginRes = await request.post('/auth/login', {
    data: { email: process.env.TEST_EMAIL, password: process.env.TEST_PASSWORD },
  });
  const { token } = await loginRes.json();

  // Create user — raw HTTP, no abstraction
  const createRes = await request.post('/users', {
    data: { name: 'Alice', email: `a-${Date.now()}@t.com` },
    headers: { 'Authorization': `Bearer ${token}` },
  });
  const user = await createRes.json();

  // Update — same HTTP noise
  const patchRes = await request.patch(`/users/${user.id}`, {
    data: { name: 'Alice Updated' },
    headers: { 'Authorization': `Bearer ${token}` },
  });
  expect(patchRes.status()).toBe(200);
});
With service layer — reads like a business scenario
test('user can update their profile', async ({ users }) => {
  // No HTTP. No tokens. No headers. Just intent.
  const user = await users.create({ name: 'Alice' });
  const updated = await users.update(user.id, { name: 'Alice Updated' });
  expect(updated.name).toBe('Alice Updated');
});

Base client — auth, error handling, shared config

Every service extends this. Auth headers, error guards, and JSON parsing live here once. Services never repeat them.

services/base.client.ts
import { APIRequestContext, APIResponse } from '@playwright/test';

export class BaseClient {
  protected readonly request: APIRequestContext;
  protected readonly baseURL: string;

  constructor(request: APIRequestContext, baseURL: string) {
    this.request = request;
    this.baseURL = baseURL;
  }

  // Central error guard — call on every response
  protected async assertOk(res: APIResponse, context: string): Promise<void> {
    if (!res.ok()) {
      const body = await res.text();
      throw new Error(
        `${context} failed — ${res.status()} ${res.url()}\n${body}`
      );
    }
  }

  // Parse JSON or return text — never throw on body read
  protected async json<T>(res: APIResponse): Promise<T> {
    try { return await res.json() as T; }
    catch { throw new Error(`Non-JSON response from ${res.url()}: ${await res.text()}`); }
  }

  // Convenience: assert + parse in one call
  protected async assertAndParse<T>(res: APIResponse, ctx: string): Promise<T> {
    await this.assertOk(res, ctx);
    return this.json<T>(res);
  }
}

The assertAndParse helper is the key. In raw tests you must remember to check the status before calling .json() every single time. In services, one method does both — the error message includes the HTTP status, URL, and body, so failures are self-describing without needing to open the Allure report.

UserService — domain operations, no HTTP noise

One method per operation. Each method is named after what it does, not the HTTP verb. Callers never see headers, status codes, or URL strings.

services/user.service.ts
import { APIRequestContext } from '@playwright/test';
import { BaseClient } from './base.client';

export interface User {
  id: number; name: string; email: string; role: string;
}
export type CreateUserDto = Omit<User, 'id'> & { password: string };
export type UpdateUserDto = Partial<Omit<User, 'id'>>;

export class UserService extends BaseClient {

  async getAll(): Promise<User[]> {
    const res = await this.request.get(`${this.baseURL}/users`);
    return this.assertAndParse<User[]>(res, 'GET /users');
  }

  async getById(id: number): Promise<User> {
    const res = await this.request.get(`${this.baseURL}/users/${id}`);
    return this.assertAndParse<User>(res, `GET /users/${id}`);
  }

  async create(data: CreateUserDto): Promise<User> {
    const email = data.email ?? `user-${Date.now()}-${Math.random().toString(36).slice(2)}@test.com`;
    const res = await this.request.post(`${this.baseURL}/users`, {
      data: { ...data, email },
    });
    return this.assertAndParse<User>(res, 'POST /users');
  }

  async update(id: number, data: UpdateUserDto): Promise<User> {
    const res = await this.request.patch(`${this.baseURL}/users/${id}`, { data });
    return this.assertAndParse<User>(res, `PATCH /users/${id}`);
  }

  async delete(id: number): Promise<void> {
    const res = await this.request.delete(`${this.baseURL}/users/${id}`);
    // 404 on delete is fine — resource may have been cleaned up already
    if (!res.ok() && res.status() !== 404) {
      await this.assertOk(res, `DELETE /users/${id}`);
    }
  }

  // Returns the raw response — for testing error cases
  async createRaw(data: Partial<CreateUserDto>) {
    return this.request.post(`${this.baseURL}/users`, { data });
  }
}

The createRaw method exists for negative tests. When testing that a 422 is returned for a missing field, you need the raw response — not a parsed object. Every service should expose both a typed convenience method (for happy-path tests) and a raw method (for error-case tests) for write operations.

PostService — same pattern, different domain

Services compose naturally — PostService can call UserService internally. Each service owns its own domain completely.

services/post.service.ts
import { APIRequestContext } from '@playwright/test';
import { BaseClient } from './base.client';

export interface Post {
  id: number; title: string; body: string; userId: number;
}
export type CreatePostDto = Omit<Post, 'id'>;

export class PostService extends BaseClient {

  async getAll(filters?: { userId?: number }): Promise<Post[]> {
    const res = await this.request.get(`${this.baseURL}/posts`, {
      params: filters,
    });
    return this.assertAndParse<Post[]>(res, 'GET /posts');
  }

  async getById(id: number): Promise<Post> {
    const res = await this.request.get(`${this.baseURL}/posts/${id}`);
    return this.assertAndParse<Post>(res, `GET /posts/${id}`);
  }

  async create(data: CreatePostDto): Promise<Post> {
    const res = await this.request.post(`${this.baseURL}/posts`, { data });
    return this.assertAndParse<Post>(res, 'POST /posts');
  }

  async update(id: number, data: Partial<CreatePostDto>): Promise<Post> {
    const res = await this.request.patch(`${this.baseURL}/posts/${id}`, { data });
    return this.assertAndParse<Post>(res, `PATCH /posts/${id}`);
  }

  async delete(id: number): Promise<void> {
    const res = await this.request.delete(`${this.baseURL}/posts/${id}`);
    if (!res.ok() && res.status() !== 404) await this.assertOk(res, `DELETE /posts/${id}`);
  }

  // Domain operation — not just a raw HTTP call
  async getAllByUser(userId: number): Promise<Post[]> {
    return this.getAll({ userId });
  }

  async createRaw(data: Partial<CreatePostDto>) {
    return this.request.post(`${this.baseURL}/posts`, { data });
  }
}

getAllByUser(userId) is an example of a domain operation — it's not just wrapping an HTTP call, it's expressing intent. A test that calls posts.getAllByUser(user.id) is immediately readable. One that calls request.get('/posts', { params: { userId: user.id } }) requires the reader to parse URL semantics to understand the same thing.

Fixtures — inject services into tests

Services are instantiated once per test via fixtures. Tests receive typed service objects — they never call new UserService().

fixtures/index.ts
import { test as base, APIRequestContext } from '@playwright/test';
import { UserService } from '../services/user.service';
import { PostService } from '../services/post.service';

type ServiceFixtures = {
  users: UserService;
  posts: PostService;
};

export const test = base.extend<ServiceFixtures>({

  users: async ({ request }, use) => {
    await use(new UserService(request, process.env.BASE_URL!));
  },

  posts: async ({ request }, use) => {
    await use(new PostService(request, process.env.BASE_URL!));
  },

});

export { expect } from '@playwright/test';

Services receive the test's request fixture — meaning they automatically use the auth headers and baseURL you configured in playwright.config.ts. No extra auth setup in services themselves. If you need a service with different auth (e.g. admin vs user), create a custom APIRequestContext and pass it instead.

Optional — pre-created resource fixtures
type ServiceFixtures = {
  users: UserService;
  posts: PostService;
  testUser: User;  // pre-created, auto-deleted
};

export const test = base.extend<ServiceFixtures>({
  users: async ({ request }, use) => {
    await use(new UserService(request, process.env.BASE_URL!));
  },
  posts: async ({ request }, use) => {
    await use(new PostService(request, process.env.BASE_URL!));
  },

  // Creates a user before the test, deletes it after
  testUser: async ({ users }, use) => {
    const user = await users.create({ name: 'Fixture User', role: 'member' });
    await use(user);                 // test runs here
    await users.delete(user.id);   // cleanup — 404 handled in service
  },
});

Tests — clean, readable, business-focused

This is what the service layer delivers. Every test reads like a specification. No HTTP, no headers, no URL strings.

// tests/users.spec.ts
import { test, expect } from '../fixtures';

test.describe('User management', () => {

  test('creates a user and fetches them back',
    { tag: '@smoke' },
    async ({ users }) => {
      const created = await users.create({ name: 'Alice', role: 'member' });
      const fetched = await users.getById(created.id);

      expect(fetched.name).toBe('Alice');
      expect(fetched.id).toBe(created.id);

      await users.delete(created.id);
    }
  );

  test('updates only the fields sent',
    { tag: '@regression' },
    async ({ users, testUser }) => {
      const updated = await users.update(testUser.id, { name: 'Updated' });

      expect(updated.name).toBe('Updated');
      expect(updated.email).toBe(testUser.email); // untouched
      expect(updated.role).toBe(testUser.role);   // untouched
    }
  );

  test('rejects missing name with 422',
    { tag: '@regression' },
    async ({ users }) => {
      // Raw method for error case — we need the response object
      const res = await users.createRaw({ role: 'member' });
      expect(res.status()).toBe(422);
      expect(await res.json()).toHaveProperty('errors');
    }
  );

});

// tests/posts.spec.ts — services compose naturally
test.describe('Post management', () => {

  test('user posts are filterable by owner',
    { tag: '@regression' },
    async ({ users, posts }) => {
      const user = await users.create({ name: 'Bob', role: 'author' });
      const post = await posts.create({ title: 'Bob post', body: 'Content', userId: user.id });

      const userPosts = await posts.getAllByUser(user.id);
      expect(userPosts.every(p => p.userId === user.id)).toBeTruthy();

      await posts.delete(post.id);
      await users.delete(user.id);
    }
  );

});

Compare this to the raw HTTP version in Tab 1. Same logic — completely different readability. A new engineer can read these tests and understand the API contract without knowing anything about HTTP, headers, or Playwright internals. That's the service layer paying off.
