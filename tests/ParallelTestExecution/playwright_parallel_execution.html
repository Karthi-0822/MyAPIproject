Playwright parallel test execution for API tests — how workers work, common pitfalls, and solutions with complete code examples

How Playwright workers run your tests

Playwright distributes tests across worker processes. Each worker is an independent Node.js process — they share nothing by default.

What each worker owns exclusively
Isolated per worker

✓ Its own Node.js process
✓ Its own memory space
✓ Its own fixture instances
✓ Its own request context
✓ Its own env variables (read-only)

Shared across all workers

✗ The database / API server
✗ Any global test data by name/email
✗ Rate limit counters
✗ External service quotas
✗ File system (if writing files)

Test distribution — how tests get assigned to workers
// With 4 workers and 12 tests, Playwright distributes like this:
// Worker 1: tests 1, 5, 9
// Worker 2: tests 2, 6, 10
// Worker 3: tests 3, 7, 11
// Worker 4: tests 4, 8, 12

// The order is not guaranteed — worker 3 might finish test 7
// before worker 1 finishes test 1. Never assume order.

Workers are processes, not threads. They truly share nothing in memory. A variable set in Worker 1 is completely invisible to Worker 2. This is good for isolation — but it means any state that does need sharing (like a cached auth token) must go through the API itself or a worker-scoped fixture.

Parallel execution config

Three settings control how parallel execution behaves. Know all three.

playwright.config.ts — parallel settings
export default defineConfig({

  // Run every test in its own worker — maximum parallelism
  // Default: false (tests in same file run sequentially)
  fullyParallel: true,

  // Number of parallel worker processes
  // Default: half the CPU count on your machine
  workers: 4,

  // Or as a percentage of CPU cores
  workers: '75%',

});
Parallelism levels — what each setting controls
SettingWhat runs in parallelWhen to use
fullyParallel: false (default)Files run in parallel — tests within a file run in orderTests in a file share state or must run in sequence

// Force this entire file to run sequentially
// even when fullyParallel: true is set globally
test.describe.configure({ mode: 'serial' });

// Or use the annotation at the file level
test.describe('stateful admin setup', () => {
  test.describe.configure({ mode: 'serial' });
  // These run in order: 1, 2, 3
  test('1. create admin user', async ({ request }) => { /* ... */ });
  test('2. assign permissions', async ({ request }) => { /* ... */ });
  test('3. verify access', async ({ request }) => { /* ... */ });
});

Pitfall 1 — shared mutable state

The most common parallel bug: two tests read, modify, or delete the same record at the same time.

broken

Two tests modify the same user

// Test A (Worker 1)
test('update user email', async ({ request }) => {
  await request.patch('/users/1', { data: { email: 'a@test.com' } });
  const user = await (await request.get('/users/1')).json();
  expect(user.email).toBe('a@test.com');
  // ❌ Worker 2 might have changed it to 'b@test.com' between patch and get
});

// Test B (Worker 2) — running at the same time
test('update user name', async ({ request }) => {
  await request.patch('/users/1', { data: { email: 'b@test.com' } });
  // ❌ Both tests race on user id 1 — results are non-deterministic
});
fixed

Each test creates and owns its own user

// Both tests create their OWN user via fixture — no sharing
test('update user email', async ({ request, testUser }) => {
  // testUser is unique to this test — Worker 2 has its own testUser
  await request.patch(`/users/${testUser.id}`, {
    data: { email: 'updated@test.com' },
  });
  const user = await (await request.get(`/users/${testUser.id}`)).json();
  expect(user.email).toBe('updated@test.com');
  // ✅ No other worker touches this user's id
});

Pitfall 2 — data collision on unique constraints

Two workers creating records with the same hardcoded name or email will collide on unique constraints. This is the most common flakiness cause in parallel API suites.

broken

Hardcoded email collides under parallel workers

// Both workers try to create this exact user simultaneously
test('create user', async ({ request }) => {
  const res = await request.post('/users', {
    data: { name: 'Test User', email: 'test@example.com' },
    // ❌ Worker 1 and Worker 2 both POST this — one gets 409 Conflict
  });
  expect(res.status()).toBe(201);
});
fixed

Unique identifier per test run

// Generate a unique email for every test invocation
const uid = () => `${Date.now()}-${Math.random().toString(36).slice(2)}`;

test('create user', async ({ request }) => {
  const res = await request.post('/users', {
    data: {
      name: 'Test User',
      email: `user-${uid()}@test.com`, // unique every time
    },
  });
  expect(res.status()).toBe(201);
  // ✅ Worker 1 and Worker 2 create different emails — no collision
});

Why both Date.now() and Math.random()? Date.now() alone can collide when two workers generate the email within the same millisecond on a fast machine. Adding the random suffix makes collision probability negligible. Use the same pattern for usernames, slugs, and any other unique-constrained field.

Pitfall 3 — assuming test execution order

Parallel tests run in an unpredictable order. Tests that depend on each other's side effects break silently.

broken

Test 2 assumes Test 1 already ran

// tests/users.spec.ts
let createdUserId: number;

test('1. create user', async ({ request }) => {
  const res = await request.post('/users', { data: { name: 'Alice' } });
  createdUserId = (await res.json()).id;
  // ❌ With fullyParallel: true, test 2 may run before this
});

test('2. fetch that user', async ({ request }) => {
  // ❌ createdUserId may still be undefined
  const res = await request.get(`/users/${createdUserId}`);
  expect(res.status()).toBe(200);
});
fixed

Each test is fully self-contained

// Option A — chain within a single test
test('create then fetch user', async ({ request }) => {
  const created = await (await request.post('/users', {
    data: { name: 'Alice', email: `a-${Date.now()}@t.com` },
  })).json();

  const fetched = await (await request.get(`/users/${created.id}`)).json();
  expect(fetched.name).toBe('Alice');
  // ✅ All steps are in one test — order is guaranteed
});

// Option B — use a fixture to provide pre-created data
test('fetch user', async ({ request, testUser }) => {
  const res = await request.get(`/users/${testUser.id}`);
  expect(res.status()).toBe(200);
  // ✅ testUser created by fixture before this test ran
});

Safe patterns for parallel API tests

These patterns make tests parallel-safe by design — not by luck.

The complete safe parallel test file
// tests/users.spec.ts — safe for fullyParallel: true
import { test, expect } from '../fixtures';

// ── Rule 1: No shared mutable variables between tests ─────
// ── Rule 2: Every test owns its data via fixtures or uid() ─
// ── Rule 3: No test assumes another test ran first ─────────

const uid = () => `${Date.now()}-${Math.random().toString(36).slice(2)}`;

test.describe('Users API', () => {

  test('POST — creates unique user', async ({ request }) => {
    const res = await request.post('/users', {
      data: { name: 'Alice', email: `alice-${uid()}@test.com` },
    });
    expect(res.status()).toBe(201);
    // Cleanup — this test created it, this test deletes it
    const { id } = await res.json();
    await request.delete(`/users/${id}`);
  });

  test('GET — reads own user', async ({ request, testUser }) => {
    // testUser fixture creates + deletes — safe across workers
    const res = await request.get(`/users/${testUser.id}`);
    expect(res.status()).toBe(200);
  });

  test('PATCH — updates own user only', async ({ request, testUser }) => {
    const res = await request.patch(`/users/${testUser.id}`, {
      data: { name: 'Updated' },
    });
    expect(res.status()).toBe(200);
    // Only touches testUser.id — invisible to other workers
  });

});
Parallel safety checklist
CheckSafe?Fix if not
Using hardcoded IDs like /users/1✗ UnsafeCreate resources in fixtures, use dynamic IDs
Sharing a let variable between tests✗ UnsafeMove state inside the test or into a fixture
Hardcoded unique email test@test.com✗ UnsafeUse user-${uid()}@test.com
Test B reads data created by Test A✗ UnsafeChain inside one test or use a fixture
Each test creates + cleans up its own data✓ Safe—
Using testUser.id from a fixture✓ Safe—
Read-only GET tests with no mutations✓ Safe—

The single rule that makes a test parallel-safe: it must not depend on data it did not create, and must clean up everything it did create. A test that creates its own unique user, uses only that user's ID throughout, and deletes it in teardown will never conflict with any other test — regardless of how many workers are running.
