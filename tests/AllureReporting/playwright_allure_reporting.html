Allure reporting with Playwright — complete setup guide covering installation, configuration, API request attachments, labels, steps, and CI integration

Install and configure

Three packages, one config change. Everything else builds on this.

Install
# The reporter and the decorator library for labels/steps/attachments
npm install -D allure-playwright allure-js-commons

# Allure CLI — generates the HTML report from raw results
npm install -D allure-commandline
playwright.config.ts — add the reporter
import { defineConfig } from '@playwright/test';

export default defineConfig({
  reporter: [
    // Allure writes raw result files here
    ['allure-playwright', {
      detail: true,            // include test steps
      outputFolder: 'allure-results', // raw result files land here
      suiteTitle: true,            // use describe block as suite name
    }],
    // Keep list reporter for local terminal output
    ['list'],
  ],
});
Add scripts to package.json
{
  "scripts": {
    "test": "playwright test",
    "allure:generate": "allure generate allure-results --clean -o allure-report",
    "allure:open": "allure open allure-report",
    "allure:serve": "allure serve allure-results"
  }
}

Two-step process. Running tests writes raw .json files into allure-results/. The allure generate command then converts those into a browsable HTML site in allure-report/. allure serve does both steps at once and opens a local server — good for local debugging.

Run tests and generate your first report

The exact commands to go from zero to a working Allure report in your browser.

1

Run the tests

Raw result files are written to allure-results/

npx playwright test
2

Generate the HTML report

Converts raw files into a self-contained HTML site

npm run allure:generate
# or directly:
npx allure generate allure-results --clean -o allure-report
3

Open the report

Starts a local server and opens in your browser

npm run allure:open
# or one command for both generate + open:
npm run allure:serve
Add allure-results to .gitignore
# .gitignore
allure-results/
allure-report/

Always use --clean when generating. Without it, Allure merges new results with old ones — a test that previously failed shows as flaky even if it now always passes. --clean wipes the output folder before generating so you always see only the latest run.

Attach API request and response details

The most valuable Allure feature for API testing — attach the full request and response to every test so you can debug failures without re-running.

tests/helpers/allure-request.ts — reusable attachment helper
import { APIResponse } from '@playwright/test';
import * as allure from 'allure-js-commons';

interface RequestLog {
  method: string;
  url: string;
  payload?: unknown;
}

export async function attachRequest(
  response: APIResponse,
  req: RequestLog
): Promise<void> {

  // Attach request details
  const requestData = {
    method: req.method,
    url: req.url,
    payload: req.payload ?? null,
  };
  await allure.attachment(
    'Request',
    JSON.stringify(requestData, null, 2),
    { contentType: 'application/json' }
  );

  // Attach response status + headers + body
  const responseData = {
    status: response.status(),
    headers: response.headers(),
    body: await safeJson(response),
  };
  await allure.attachment(
    'Response',
    JSON.stringify(responseData, null, 2),
    { contentType: 'application/json' }
  );
}

async function safeJson(r: APIResponse): Promise<unknown> {
  try { return await r.json(); }
  catch { return await r.text(); }
}
Using the helper in a test
import { test, expect } from '@playwright/test';
import { attachRequest } from './helpers/allure-request';

test('POST /users creates a user', async ({ request }) => {
  const payload = { name: 'Alice', email: `a-${Date.now()}@test.com` };

  const response = await request.post('/users', { data: payload });

  // Attach full request + response to the Allure report
  await attachRequest(response, {
    method: 'POST',
    url: '/users',
    payload,
  });

  expect(response.status()).toBe(201);
});

Always attach before asserting. If you attach after the assertion and the assertion throws, the attachment never runs — you lose the response body right when you need it most. Attach immediately after the request returns, then assert.

Labels, severity, and metadata

Allure metadata makes reports filterable and readable for non-engineers. Add these to every test.

All available decorators
import * as allure from 'allure-js-commons';

test('GET /users returns user list', async ({ request }) => {

  // Suite hierarchy — appears in Allure's tree navigation
  await allure.epic('User management');
  await allure.feature('User listing');
  await allure.story('Admin can view all users');

  // Severity — critical, normal, minor, trivial, blocker
  await allure.severity('critical');

  // Tags — same as Playwright @tags, visible in report
  await allure.tag('smoke');
  await allure.tag('users');

  // Link to ticket / docs
  await allure.issue('https://jira.yourco.com/browse/API-42');
  await allure.tms('https://testrail.yourco.com/C123');

  // Owner — visible in Allure dashboard
  await allure.owner('alice@yourco.com');

  // Description — rendered as markdown in report
  await allure.description(`
    Verifies GET /users returns a non-empty array.
    Each item must have id, name, and email.
  `);

  const response = await request.get('/users');
  expect(response).toBeOK();
});
Named steps — visible in the report timeline
test('full user flow with named steps', async ({ request }) => {
  await allure.severity('critical');

  // allure.step() wraps a block — shows in the report timeline
  const user = await allure.step('Create test user', async () => {
    const res = await request.post('/users', {
      data: { name: 'Alice', email: `a-${Date.now()}@t.com` },
    });
    await attachRequest(res, { method: 'POST', url: '/users' });
    return res.json();
  });

  await allure.step('Verify user is fetchable', async () => {
    const res = await request.get(`/users/${user.id}`);
    await attachRequest(res, { method: 'GET', url: `/users/${user.id}` });
    expect(res.status()).toBe(200);
  });

  await allure.step('Clean up — delete user', async () => {
    await request.delete(`/users/${user.id}`);
  });
});

Complete real-world test file

All patterns together — metadata, steps, request attachments, and assertions in one production-ready spec.

// tests/users.spec.ts
import { test, expect } from '@playwright/test';
import * as allure from 'allure-js-commons';
import { attachRequest } from './helpers/allure-request';

test.describe('Users API', () => {

  test('POST /users — creates and persists a user',
    { tag: ['@smoke', '@regression'] },
    async ({ request }) => {

      // Metadata
      await allure.epic('User management');
      await allure.feature('Create user');
      await allure.severity('critical');
      await allure.owner('backend-team');

      const payload = {
        name: 'Alice',
        email: `alice-${Date.now()}@test.com`,
      };

      // Step 1 — create
      const created = await allure.step('POST /users', async () => {
        const res = await request.post('/users', { data: payload });
        await attachRequest(res, { method: 'POST', url: '/users', payload });
        expect(res.status()).toBe(201);
        return await res.json();
      });

      // Step 2 — verify persistence
      await allure.step('GET /users/:id — verify persisted', async () => {
        const res = await request.get(`/users/${created.id}`);
        await attachRequest(res, { method: 'GET', url: `/users/${created.id}` });
        expect(res.status()).toBe(200);
        const fetched = await res.json();
        expect(fetched.email).toBe(payload.email);
      });

      // Step 3 — clean up
      await allure.step('DELETE /users/:id', async () => {
        const res = await request.delete(`/users/${created.id}`);
        expect(res.status()).toBe(204);
      });
    }
  );

});

CI workflow — generate and publish the report

Run tests, generate the report, and upload it as an artifact — all in one GitHub Actions workflow.

name: API tests with Allure

on:
  pull_request:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Run Playwright tests
        run: npx playwright test
        env:
          CI: true
          BASE_URL: ${{ secrets.STAGING_BASE_URL }}
          API_TOKEN: ${{ secrets.STAGING_API_TOKEN }}

      - name: Generate Allure report
        if: always()
        run: npx allure generate allure-results --clean -o allure-report

      - name: Upload Allure report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: allure-report-${{ github.run_id }}
          path: allure-report/
          retention-days: 30

       # Optional: also upload raw results for history tracking
      - name: Upload raw results
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: allure-results-${{ github.run_id }}
          path: allure-results/
          retention-days: 7

Upload both allure-report/ and allure-results/. The generated report is what you download and open in a browser right now. The raw results are what Allure uses to build trend charts across runs — if you connect Allure to GitHub Pages or Allure TestOps, it reads the raw results history to show pass-rate trends over time.
