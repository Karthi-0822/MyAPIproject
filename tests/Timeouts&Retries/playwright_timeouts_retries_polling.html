Timeouts, retries, and polling in Playwright API tests — complete guide with real examples for each pattern

Timeouts — three levels, three jobs

Playwright has three separate timeout settings. Each covers a different scope. Knowing which to set where eliminates most "test timed out" confusion.

TimeoutDefaultControlsWhere to set
timeout30 000msEntire test — all steps combinedConfig or per-test
actionTimeout0 (inherit)Single request or actionConfig use:{}
navigationTimeout30 000msPage navigation (UI tests)Config use:{}
Global timeouts in playwright.config.ts
export default defineConfig({
  // Each test has 30s total to complete all its requests
  timeout: 30_000,

  use: {
    // Each individual request times out after 10s
    actionTimeout: 10_000,
  },
});
Per-request timeout — override for one slow call
test('slow report generation endpoint', async ({ request }) => {
  // This endpoint is known to take up to 45s — set per-request
  const response = await request.post('/reports/generate', {
    data: { type: 'annual', year: 2024 },
    timeout: 45_000, // ms — this request only
  });
  expect(response.status()).toBe(200);
}, { timeout: 60_000 }); // overall test timeout too
Per-test timeout — for a whole describe group
test.describe('slow integration flows', () => {
  // Every test in this group gets 60s
  test.describe.configure({ timeout: 60_000 });

  test('full signup flow', async ({ request }) => { /* ... */ });
  test('data export flow', async ({ request }) => { /* ... */ });
});

Don't raise global timeouts to fix flaky tests. A flaky test hiding behind a 120s timeout is still a flaky test — it just takes twice as long to fail. Timeouts are a diagnostic tool: if a test consistently needs 45s, that request is the problem, not the timeout value. Raise only the timeout for the specific request that needs it.

Test retries — Playwright's built-in retry

Playwright can automatically re-run a failed test. This is for genuinely flaky infrastructure — network blips, cold-start latency — not for hiding bugs.

Global retry config
export default defineConfig({
  // Retry failed tests up to 2 times in CI, never locally
  retries: process.env.CI ? 2 : 0,
});

Retries should be 0 locally and 1–2 in CI. Retrying locally hides bugs — you fix the test by waiting for it to pass on a lucky run. Retrying in CI catches genuine infrastructure flakiness (rate limits, network timeouts, cold starts) without masking code bugs, because you see the retry count in the report.

Per-test retry — for one known-flaky test
test('webhook delivery — eventually consistent',
  { retries: 3 }, // this test only
  async ({ request }) => {
    const res = await request.get('/webhooks/latest');
    expect(res.status()).toBe(200);
    expect((await res.json()).delivered).toBe(true);
  }
);
test.info().retry — detect which attempt you're on
test('with retry-aware setup', async ({ request }) => {
  const attempt = test.info().retry; // 0 = first run, 1 = first retry

  if (attempt > 0) {
    console.log(`Retry attempt ${attempt} — clearing cache first`);
    await request.delete('/cache/test-data');
  }

  const res = await request.get('/data');
  expect(res).toBeOK();
});

Retries re-run the entire test from the beginning — including fixtures. If your fixture creates a user via API, it will create another user on retry. Make sure your fixture teardown handles this (the 404 guard on delete) and that your test data uses unique identifiers like Date.now().

Polling — waiting for async operations

Some operations don't complete instantly — jobs, exports, emails, webhooks. Polling repeatedly calls an endpoint until the expected state appears.

waitForCondition — reusable polling helper
// tests/helpers/poll.ts
import { APIRequestContext } from '@playwright/test';

interface PollOptions {
  interval?: number; // ms between attempts — default 1000
  timeout?: number; // max ms to wait — default 30000
  message?: string; // error message on timeout
}

export async function pollUntil<T>(
  fn: () => Promise<T>,
  condition: (result: T) => boolean,
  opts: PollOptions = {}
): Promise<T> {
  const { interval = 1000, timeout = 30_000, message = 'Polling timed out' } = opts;
  const deadline = Date.now() + timeout;

  while (Date.now() < deadline) {
    const result = await fn();
    if (condition(result)) return result;
    await new Promise(r => setTimeout(r, interval));
  }
  throw new Error(message);
}
Using the helper — wait for a background job
import { test, expect } from '@playwright/test';
import { pollUntil } from './helpers/poll';

test('export job completes within 30s',
  { timeout: 40_000 }, // test timeout > polling timeout
  async ({ request }) => {

    // Step 1 — kick off the job
    const startRes = await request.post('/exports', {
      data: { format: 'csv', dateRange: '2024' },
    });
    expect(startRes.status()).toBe(202); // 202 = accepted, not done yet
    const { jobId } = await startRes.json();

    // Step 2 — poll until status is 'completed'
    const job = await pollUntil(
      async () => (await request.get(`/exports/${jobId}`)).json(),
      result => result.status === 'completed',
      {
        interval: 2000,
        timeout: 30_000,
        message: `Export job ${jobId} did not complete in 30s`,
      }
    );

    // Step 3 — assert the completed result
    expect(job.status).toBe('completed');
    expect(job.downloadUrl).toMatch(/\.csv$/);
  }
);

202 Accepted is the correct status for async operations. It means "I received the request and started working on it — come back later to check." 200 or 201 implies the work is done. If your endpoint returns 200 for a job that isn't finished yet, that's a contract bug worth noting.

Retry wrapper — retry a single request, not the whole test

Sometimes one flaky request inside a stable test needs retrying. A request-level retry wrapper handles this without touching the rest of the test.

retryRequest — retry one call with backoff
// tests/helpers/retry.ts
import { APIResponse, APIRequestContext } from '@playwright/test';

interface RetryOptions {
  attempts?: number;       // total attempts — default 3
  delay?: number;       // ms between attempts — default 1000
  retryOn?: number[];     // status codes that trigger a retry
}

export async function retryRequest(
  fn: () => Promise<APIResponse>,
  opts: RetryOptions = {}
): Promise<APIResponse> {
  const { attempts = 3, delay = 1000, retryOn = [429, 503, 504] } = opts;

  for (let i = 0; i < attempts; i++) {
    const response = await fn();

    // Success or non-retryable error — return immediately
    if (!retryOn.includes(response.status())) return response;

    // Last attempt — return whatever we got
    if (i === attempts - 1) return response;

    // Exponential backoff: 1s, 2s, 4s...
    const wait = delay * Math.pow(2, i);
    console.log(`Retry ${i + 1}/${attempts - 1} after ${wait}ms (status ${response.status()})`);
    await new Promise(r => setTimeout(r, wait));
  }
  throw new Error('retryRequest: unreachable');
}
Using the retry wrapper
import { retryRequest } from './helpers/retry';

test('GET /data — handles rate limiting gracefully', async ({ request }) => {

  // Retry on 429 (rate limit) and 503 (service unavailable)
  const response = await retryRequest(
    () => request.get('/data'),
    { attempts: 4, delay: 500, retryOn: [429, 503] }
  );

  expect(response.status()).toBe(200);
  expect(await response.json()).toHaveProperty('data');
});

Retry on infrastructure errors only — 429, 503, 504. Never retry on 400, 401, 403, 404, or 422. Those are deterministic errors — retrying them wastes time and hides bugs. The server will return the same error every time. Retry only on errors that are caused by transient conditions: rate limits, service unavailability, and gateway timeouts.

Full production patterns

All three tools — timeouts, retries, and polling — working together in real test scenarios.

Pattern 1 — webhook delivery test
// Trigger an action, wait for its async side effect to appear
test('payment webhook is delivered within 10s',
  { timeout: 20_000, tag: ['@regression', '@slow'] },
  async ({ request }) => {

    // Fire the payment
    const payRes = await request.post('/payments', {
      data: { amount: 1000, currency: 'USD' },
    });
    const { paymentId } = await payRes.json();

    // Poll the webhook log until delivery is confirmed
    const webhook = await pollUntil(
      async () => (await request.get(`/webhooks?paymentId=${paymentId}`)).json(),
      result => result.delivered === true,
      { interval: 1000, timeout: 10_000, message: 'Webhook not delivered in 10s' }
    );

    expect(webhook.delivered).toBe(true);
    expect(webhook.paymentId).toBe(paymentId);
  }
);
Pattern 2 — third-party API with rate limit handling
test('syncs data from rate-limited third-party API',
  { timeout: 30_000 },
  async ({ request }) => {

    // Retry the external call — it rate-limits at 10 req/s
    const syncRes = await retryRequest(
      () => request.post('/sync/external', { timeout: 10_000 }),
      { attempts: 3, delay: 2000, retryOn: [429] }
    );
    expect(syncRes.status()).toBe(202);
    const { syncId } = await syncRes.json();

    // Then poll until sync completes
    const result = await pollUntil(
      async () => (await request.get(`/sync/${syncId}`)).json(),
      r => ['completed', 'failed'].includes(r.status),
      { interval: 3000, timeout: 20_000 }
    );

    expect(result.status).toBe('completed');
    expect(result.recordsSynced).toBeGreaterThan(0);
  }
);

In the poll condition above, checking for 'completed' OR 'failed' is important. If you only poll until 'completed' and the job fails, the polling runs for the full timeout period before throwing — adding 20s of wasted time to every failed test run. Exit polling on any terminal state, then assert which state you got.
